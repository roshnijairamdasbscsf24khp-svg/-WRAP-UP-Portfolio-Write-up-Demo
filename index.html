03

Challenges Faced & How I Solved Them

🐛 Quiz App — Inline onclick handlers causing classList errors on null elements
When the quiz card's innerHTML was replaced for the next question, inline onclick handlers were firing against stale DOM references, causing "Cannot read properties of null (reading 'classList')". Solved by replacing all inline onclick attributes with addEventListener calls attached after each innerHTML update — ensuring the DOM elements always exist before any event listener touches them.
🐛 Quiz options showing blank text for HTML tag answers
Options like <style>, <link>, and <h1> were injected raw into innerHTML, so the browser interpreted them as actual HTML tags and rendered them invisibly. Fixed by escaping angle brackets as &lt; and &gt; so they display as text rather than being parsed as markup.
📱 Getting HTML files to open correctly on mobile
Files referencing separate CSS and JS files failed on mobile because relative paths don't resolve correctly when opening local files. Solved by consolidating every project into a single self-contained .html file with all CSS in a <style> block and all JavaScript in a <script> block — eliminating all external dependencies.
🛒 Cart state resetting on page navigation in the capstone
Since the capstone uses a single-page architecture with JavaScript-controlled page switching, cart state was lost between renders. Solved by syncing the cart array to localStorage on every update and reading from it on load — giving the cart persistence across sessions and page changes.
04

200-Word Reflection

Starting this internship, I could write basic HTML and style pages with CSS. Four weeks later, I have built eight complete projects — from a simple landing page to a fully functional multi-page e-commerce website with a working cart, checkout validation, and real-time search.

The most valuable lesson was learning to debug. Every project broke in a new way. The quiz app had DOM timing bugs. The countries app had CORS and API error cases to handle. The capstone required thinking about state management, localStorage, and how a single JavaScript file can power an entire website without any backend. Each problem forced me to understand the code more deeply rather than just copying and pasting.

Working at QWETRUM TECHNOLOGIES gave me real deadlines and real expectations. There was no hand-holding — I had to research, break things, fix them, and deliver. That pressure made me grow faster than any tutorial could.

Looking back at Week 1 versus Week 4, the difference is not just in the code — it's in the confidence to open a blank file and know I can build something worth showing. That is the real result of this internship.

05

2 Technologies I Want to Learn Next

⚛️

React.js

After building everything in vanilla JavaScript, I can see exactly why a component-based framework like React exists. Managing state, re-rendering only what changes, and building reusable UI components would have made the capstone's page-switching architecture far cleaner and more scalable. React is the natural next step.

🟢

Node.js & Express

Every project I built was frontend-only. I used localStorage as a substitute for a real database, and hardcoded dummy product data in JavaScript arrays. Learning Node.js would let me build actual APIs, connect a real database, handle user authentication, and turn these static sites into full-stack applications.
